## 测试方法

### 单元测试策略

所有规则测试基于 `@parser.parse_string()` 构造 AST 进行验证，无需依赖真实文件系统：

```moonbit
test "CWE-116: replace with HTML metachar should trigger" {
  let source = "fn foo(s : String) -> String { s.replace(old=\"<\", new=\"&lt;\") }"
  let (impls, _) = @parser.parse_string(source, name="test.mbt")
  let findings : Array[Finding] = []
  run_cwe116(impls, "test.mbt", source, findings, Error)
  assert_true(findings.length() > 0)
}
```

这种方式确保：
- 测试隔离性：不依赖外部文件或环境
- 可重复性：同一输入始终产生同一结果
- 跨平台：wasm、js、native 编译目标下行为一致

### 集成测试

通过 CLI 对真实项目进行端到端扫描验证：

```bash
# 扫描真实项目
moon run src/main -- /path/to/mocket

# LLM 分析流程
moon run src/main -- llm-analyze --format script /path/to/mocket
python3 llm_analyze.py

# PoC 生成
moon run src/main -- generate-poc /path/to/mocket

# 修复建议
moon run src/main -- remediate /path/to/mocket

# 统计报告
moon run src/main -- summary /path/to/mocket
```

### CI 持续集成

GitHub Actions 配置（`.github/workflows/check.yml`）在每次推送时自动运行：

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, macos-latest, windows-latest]

steps:
  - moon check          # 编译检查
  - moon test           # 运行全部测试
  - moon fmt && git diff --exit-code   # 格式一致性
  - moon info && git diff --exit-code  # 接口文件一致性
```

### Web 规则人工审查详情

**mocket (12/12 真阳性)**：
- `mongoose.mbt:190` — `res.set_header(key, value)` 遍历 dispatch 结果直接写入，CRLF 可注入（CWE-113）
- `mongoose.mbt:195` — Set-Cookie 头写入 `cookie.to_string()`，Cookie value 含 CRLF 可分割响应（CWE-113）
- `poc_server/main.mbt:11` — `html("<h1>Hello, \{name}!</h1>")` 字符串插值含用户输入（CWE-79/template）
- 3 个 `set_cookie()` 缺少 http_only/secure/same_site（CWE-614）
- 4 个 `handle_cors()` 无显式 origin 调用（CWE-942）
- 2 个 `abort()` 调用（CWE-248）

**luna.mbt (CWE-113, 3/14 真阳性)**：
- `route_params.mbt:238` — `PageProps::set_header(name, value)` 直接透传任意 header（真阳性）
- `mbtx_handler.mbt:35` — `content_type` 来自 JS FFI 返回值，外部可控（真阳性）
- `astra/middleware.mbt:90` — 动态 content_type 设置（真阳性）
- 其余 11 个为服务端 config 对象驱动的 header 设置，值非用户可控（假阳性）

**cmark.mbt (CWE-79, 1/1 真阳性)**：
- `render()` 显式 `safe=false`，`<script>` 和 `javascript:` URL 直接输出

**crescent (13/13 真阳性)**：Cookie(3)/DoS(5)/CORS(4)/abort(1)

**Web 规则整体真阳率**：去除 luna.mbt config 驱动假阳性后 100% (26/26)；含 luna.mbt 为 70% (26/37)。

### 本地动态验证环境搭建

所有 PoC 均需本地部署目标项目后验证。完整流程：

**1. 环境要求**

- MoonBit 工具链 (moon v0.1.20260703+)：`curl -fsSL https://cli.moonbitlang.com/install/unix.sh | bash`
- Python 3.8+（标准库，无第三方依赖）
- curl、netcat（可选）

**2. 克隆目标项目并构建**

```bash
mkdir -p /tmp/audit-targets && cd /tmp/audit-targets
git clone https://github.com/bobzhang/crescent.git
git clone https://github.com/oboard/mocket.git
git clone https://github.com/moonbitlang/async.git
git clone https://github.com/moonbit-community/cmark.mbt.git

for proj in crescent mocket async cmark.mbt; do
  cd /tmp/audit-targets/$proj && moon build --target native 2>&1 | tail -1
  cd /tmp/audit-targets
done
```

**3. 启动靶机 → 运行 PoC → 判定结果**

每个项目先用 `moon run examples/<server> --target native &` 启动后台服务，再运行对应 Python PoC 脚本验证。PoC 脚本输出 CONFIRMED / FAILED 判定结果。

详细的端到端操作步骤见 [red-team-poc-report.md](../../red-team-poc-report.md) 中的「本地环境搭建与 PoC 验证」章节。
