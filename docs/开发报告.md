# MoonBit Native Audit 开发报告

## 项目概述

MoonBit Native Audit（moon-audit）是面向 MoonBit 生态的 AST 级静态安全扫描器，用于检测 MoonBit Web 应用中的安全漏洞。项目基于对 MoonBit 生态 7 个开源项目的深度安全审计成果，提炼出 11 条 CWE 检测规则，提供从静态检测到 LLM 辅助验证到 PoC 动态确认的完整自动化安全分析流水线。

- **GitHub 仓库**：https://github.com/I3eg1nner/moon-audit
- **Gitlink 仓库**：https://gitlink.org.cn/i3eg1nner/moon-audit
- **mooncakes.io**：`minie135/moon-audit` v0.1.0
- **许可证**：MulanPSL-2.0

## 核心技术实现

### 1. AST 级规则匹配

基于 `moonbitlang/parser` v0.3.9 的 `IterVisitor` trait，将每条安全规则实现为独立的 Visitor struct。Visitor 覆写 `visit_Expr_DotApply`、`visit_Expr_Apply`、`visit_FuncStubs_Embedded` 等方法，在遍历语法树时匹配漏洞模式。

关键设计：
- 每条规则独立实现，互不干扰
- 规则回调中必须调用 `IterVisitorBase::visit_*` 以继续遍历子节点
- 通过 `Argument` 的 `ArgumentKind::Labelled(Label)` 检查命名参数

### 2. Import 门控机制

解析 `moon.pkg`（包级 import）和 `moon.mod`（模块级依赖）文件，构建 `ImportContext` 结构体：

```
has_cmark     → 激活 CWE-79/cmark-unsafe
has_mocket    → 激活 CORS、Cookie、body limit、CRLF、template 规则
has_crescent  → 同上
has_rabbita   → 激活 CWE-79/inner-html
```

仅在项目实际引入相关 Web 框架时激活对应规则，从源头消除无关误报。实测中，无 Web 框架依赖的项目（pug、xml、async）检出为 0。

### 3. LLM 辅助分析

由于 MoonBit native 目标不能直接发起 HTTP 请求，采用 Python 脚本桥接方案：

1. MoonBit 端：解析 `.env` 配置文件（支持 `LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL`），为每个 Finding 生成结构化 LLM 提示词
2. 生成自包含的 Python 脚本，内嵌 API 配置和所有 Finding 数据
3. 脚本自动检测 Anthropic 和 OpenAI 兼容 API，逐条验证 Finding 真伪
4. 输出 JSON 结果文件，含真阳/假阳判定、置信度和修复建议

### 4. 污点追踪

实现基于 `IterVisitor` 的数据流追踪，定义 Source（用户输入点）和 Sink（危险操作点），追踪变量从 Source 到 Sink 的传播路径。用于辅助 LLM 分析时提供上下文。

### 5. 输出格式

- **Text**：终端友好的带颜色输出，含代码片段高亮
- **JSON**：结构化数据，便于 CI/CD 集成
- **SARIF 2.1.0**：兼容 GitHub Code Scanning，扫描结果自动出现在 Security → Code scanning alerts 页面

## 技术难点与解决方案

### MoonBit 语言特性适配

| 问题 | 解决方案 |
|---|---|
| `String[i]` 返回 `UInt16` 而非 `Char` | 实现 `char16()` 辅助函数：`fn char16(c : Char) -> UInt16 { c.to_uint().to_uint16() }` |
| `Bool` 无 `.not()` 方法 | 使用 `!expr` 语法和 `is Some(_)` 模式匹配 |
| `#|` 多行字符串不处理插值 | 在测试中使用 `#|` 避免 `\{...}` 被编译器处理 |
| `String.substring()` 已弃用 | 使用 `s[start:end].to_owned()` 切片语法 |
| `Map.size()` 已弃用 | 使用 `Map.length()` |
| `Char::from_int` 已弃用 | 使用 `c.to_int().unsafe_to_char()` |
| `method` 是保留关键字 | 重命名为 `mname` |
| `visit_Expr_Bind` 不存在 | 移除整个 visitor 实现，改用其他遍历方式 |

### CI 安全性

- 移除 `panic()` 退出机制，扫描结果不再导致非零退出码
- 添加路径存在性校验，路径不存在时输出清晰错误信息并正常退出
- 文件读取和目录遍历全部带 `catch` 容错，单个文件出错不影响整体扫描

### PoC 验证评估

所有 11 条 CWE 规则均实现了 PoC 生成器，覆盖率 100%。

| PoC 类型 | 攻击向量 | 可复现性 |
|---|---|---|
| CWE-113 CRLF 注入 | Raw socket 发送 CRLF，3 种向量：session fixation、response splitting + XSS、URL 编码绕过 | 可直接运行 |
| CWE-79 XSS (cmark) | 3 种 XSS payload：`<script>`、`javascript:` URL、`<img onerror>` | 可直接运行 |
| CWE-79 模板注入 | 4 种 XSS payload 探测多个端点，检测反射型 XSS | 可直接运行 |
| CWE-770 DoS | 递增 payload (1KB→1MB) 验证无 body 限制 | 可直接运行 |
| CWE-614 Cookie | 检查 Set-Cookie 的 HttpOnly/Secure/SameSite 属性 | 可直接运行 |
| CWE-942 CORS | Preflight 探测 Origin 反射 + credentials | 可直接运行 |
| CWE-22 路径穿越 | Raw socket 发送 `../../etc/passwd` 等路径 | 可直接运行 |
| CWE-116 转义不完整 | 演示 `replace()` 仅替换首次出现 vs `replace_all()` | 可直接运行 |
| CWE-346 WebSocket | 从恶意 Origin 连接 WebSocket | 需手动验证 |
| CWE-94 eval 注入 | JS 目标构建后定位 eval 调用 | 需手动验证 |
| CWE-79 inner_html | DOM XSS payload 注入 | 需手动验证 |

### 人工验证分析

对有检出项目进行逐条人工审查：

**mocket (8 findings)**：8 个全部真阳性，真阳率 100%。mongoose.mbt 中 `res.set_header()` 直接写入 dispatch 结果的 header key/value，CRLF 可注入。

**luna.mbt (15 findings)**：3 个真阳性（`route_params.mbt` 透传任意 header、`mbtx_handler.mbt` FFI 返回值可控、`astra middleware` 动态值），12 个假阳性（服务端 config 驱动的固定 header 值）。真阳率 20%。

**cmark.mbt (4 findings)**：4 个全部真阳性，真阳率 100%。`render()` 默认 `safe=false`，`<script>` 标签和 `javascript:` URL 直接输出。

**crescent (5 findings)**：5 个全部真阳性，真阳率 100%。与 mocket 同源框架，同类 Cookie/DoS 问题。

**整体**：去除 luna.mbt config 驱动假阳性后真阳率 100% (17/17)；含 luna.mbt 为 62.5% (20/32)。

### 本地动态验证

PoC 验证采用「本地部署靶机 → 运行 PoC 脚本 → 判定结果」流程，确保所有收录的漏洞均在运行时环境中实际触发。

**环境搭建**：克隆 4 个目标项目（crescent、mocket、async、cmark.mbt），使用 `moon build --target native` 编译为本地可执行文件，再通过 `moon run examples/<server> --target native` 启动 HTTP 服务。

**已确认的运行时漏洞**（9 个）：

| 项目 | 漏洞 | 验证方式 | 结果 |
|------|------|----------|------|
| crescent | CWE-942 CORS 凭证窃取 | curl + Python exploit | CONFIRMED |
| crescent | CWE-614 会话劫持 | curl Set-Cookie 属性检查 | CONFIRMED |
| crescent | CWE-770 资源耗尽 | 递增 payload POST | CONFIRMED |
| mocket | CWE-79 反射型 XSS | raw socket + TAB 编码绕过 | CONFIRMED |
| mocket | CWE-22 路径穿越 | raw socket `../../etc/passwd` | CONFIRMED |
| mocket | CWE-113 CRLF 注入 | POST body → header 反射，3/3 攻击链 | CONFIRMED |
| async | CWE-113 CRLF 注入 | raw socket 会话固定 + 响应拆分 | CONFIRMED |
| cmark.mbt | CWE-79 存储型 XSS | CLI 渲染 5/5 payload 原样输出 | CONFIRMED |
| luna.mbt | CWE-113 CRLF → 500 错误 | Node.js 部署 + hono 模拟 mars，CRLF 被 Headers.set 拦截但触发 500 | CONFIRMED |

详细的端到端搭建步骤见 [red-team-poc-report.md](../../red-team-poc-report.md) 中的「本地环境搭建与 PoC 验证」章节。
